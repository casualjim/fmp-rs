## Context

The FMP client currently supports only JSON endpoints via `get_json` on `FmpHttpClient`. All JSON methods are generated by the `define_api_trait!` macro, which hardcodes `get_json` calls and returns `impl Future<Output = FmpResult<T>>`.

FMP's bulk endpoints return CSV files (Content-Type: text/csv) representing full-market datasets. These files are 10–20 MB. Eagerly materializing them into `Vec<T>` would be wasteful; callers typically process records one at a time (insert into DB, filter, aggregate). The existing `get_json` also has a minor inefficiency: it reads the response body as a `String` before parsing, which performs an unnecessary UTF-8 allocation.

The current retry middleware (`reqwest-retry` with `ExponentialBackoff`) retries 429 responses but ignores `Retry-After` headers. For bulk endpoints rate-limited to once per 60 seconds, this means retries fire too early and fail again. FMP documents a 60-second rate limit for `/profile-bulk` and `/etf-holder-bulk`; whether FMP sends a `Retry-After` header on 429 is unverified, but implementing correct `Retry-After` handling is the right behavior regardless.

Existing relevant code:
- `src/types/bulk.rs` — all 18 return types + all param types already defined (`PartParams`, `YearPeriodParams`, `EarningsSurpriseParams`, `EodParams`)
- `src/client.rs` — `get_json`, `url_with_api_key`, `map_api_error`, `make_reqwest_client`
- `src/macros.rs` — `define_api_trait!` generates JSON trait impls
- `src/errors.rs` — `FmpError` enum (needs a CSV variant)

## Goals / Non-Goals

**Goals:**
- Stream CSV responses record-by-record without buffering the full body
- Surface the HTTP error eagerly (before the stream), parse errors per-record
- Fix `get_json` to avoid the unnecessary String allocation on the success path
- Respect `Retry-After` headers on 429 responses across all endpoints
- Keep the macro-generated API pattern consistent with existing endpoints
- CLI bulk commands with `--all` for part-based endpoints, paced to respect the 60s rate limit
- Zero breaking changes to existing public API (except `PartParams.part: String` → `u8`)

**Non-Goals:**
- Automatic pagination in the library (caller drives part iteration)
- Proactive rate-limit tracking (we react to 429, we don't predict it)
- CSV output for non-bulk endpoints
- Caching or deduplication of bulk responses

## Decisions

### 1. Return type: `FmpResult<impl Stream<Item = FmpResult<T>>>`

**Decision**: The outer `FmpResult` covers the HTTP request (network error, non-2xx status). The inner `FmpResult<T>` covers per-record CSV deserialization. Callers `await` to get the stream, then `stream.next().await` per record.

**Alternatives considered:**
- `impl Stream<Item = FmpResult<T>>` only — can't distinguish request failure from parse failure without inspecting the first item.
- `FmpResult<Vec<T>>` — defeats the purpose; eagerly materializes the full response.

### 2. CSV library: `csv-async`

**Decision**: Use `csv-async` (async CSV reader with serde support) combined with `tokio-util`'s `StreamReader` to bridge reqwest's `bytes_stream()` to `AsyncRead`.

```
resp.bytes_stream()
  → StreamReader (tokio_util::io)    [Stream<Bytes> → AsyncRead]
  → csv_async::AsyncDeserializer     [AsyncRead → Stream<T>]
```

**Alternatives considered:**
- Sync `csv` crate in `spawn_blocking` — thread-pool overhead and complexity for no benefit.
- Manual CSV parsing — fragile, no serde integration.

### 3. Separate `define_csv_api_trait!` macro

**Decision**: Add a new `define_csv_api_trait!` macro rather than extending `define_api_trait!`. The stream return type is different enough that a single macro with mode flags would be harder to reason about.

**Alternatives considered:**
- Extend `define_api_trait!` with `[csv]` annotation — possible but complicates the existing macro.
- Hand-write the bulk trait — verbose, inconsistent with the established pattern.

### 4. Fix `get_json` to use `bytes()` + `from_slice`

**Decision**: Change `resp.text().await?` + `from_str` to `resp.bytes().await?` + `from_slice`. `map_api_error` changes from `&str` to `&[u8]`, using `String::from_utf8_lossy` on the error path only.

```rust
// before
let text = resp.text().await?;
if !status.is_success() { return Err(map_api_error(status, &text)); }
serde_json::from_str(&text)?

// after
let bytes = resp.bytes().await?;
if !status.is_success() { return Err(map_api_error(status, &bytes)); }
serde_json::from_slice(&bytes)?
```

### 5. Custom `RetryAfterPolicy` for 429 handling

**Decision**: Implement a custom `reqwest_retry::RetryPolicy` in `src/retry.rs` that checks for a `Retry-After` header on 429 responses and sleeps the indicated duration before retrying. Falls back to exponential backoff when no `Retry-After` is present. Replaces `ExponentialBackoff` in `make_reqwest_client`.

`Retry-After` can be:
- An integer number of seconds: `Retry-After: 60`
- An HTTP-date: `Retry-After: Wed, 21 Oct 2026 07:28:00 GMT`

The policy handles both forms. On non-429 transient errors (5xx, 408, network failures) it delegates to exponential backoff.

**Alternatives considered:**
- Keep `ExponentialBackoff` and just increase max delay — doesn't respect server guidance, may still retry too early or wait too long.
- Patch `reqwest-retry` upstream — slower, this is a clean implementation within our codebase.

### 6. Remove default `Content-Type`; add per-request `Accept` headers

**Decision**: Remove `Content-Type: application/json` from the default headers in `make_reqwest_client`. `Content-Type` describes a request body — it is semantically meaningless on GET requests and wrong for CSV endpoints. Instead, set `Accept` on each request at the call site:

- `get_json` → `Accept: application/json`
- `get_csv` → `Accept: text/csv`

This makes each method self-describing and removes a misleading default that would otherwise bleed onto bulk requests. `text/csv` is the correct media type for CSV; `application/octet-stream` is only a generic binary fallback and is not appropriate here.

**Alternatives considered:**
- Override `Content-Type` per bulk request — still wrong (no request body); `Accept` is the correct header for negotiating response format.
- Keep the default and do nothing — bulk endpoints would receive `Content-Type: application/json` on every GET, which is incorrect and may affect server behavior.

### 7. `PartParams.part: u8`

**Decision**: Change from `String` to `u8`. FMP documents exactly 4 parts (0–3) for profile-bulk and etf-holder-bulk. `u8` is the correct type; `serde_urlencoded` serializes it as the decimal integer string, which is what the API expects.

### 7. CLI `--all` flag for part-based endpoints

**Decision**: The CLI adds `--all` to the `profile-bulk` and `etf-holder-bulk` subcommands. When set, it fetches all 4 parts sequentially. Between parts, it tracks elapsed time since the previous request started and sleeps `max(0, 60s − elapsed)` before issuing the next request — proactively respecting the rate limit rather than waiting for a 429.

```
part 0 started at T=0   → streams records → done at T=45s → sleep 15s
part 1 started at T=60s → streams records → done at T=90s → sleep 30s
part 2 started at T=120s → ...
part 3 started at T=180s → ...
```

Without `--all`, the CLI takes `--part <0-3>` and fetches a single part.

## Risks / Trade-offs

- **CSV header casing**: `csv-async` with serde uses CSV header row names as field identifiers. FMP's CSV headers must be camelCase to match `#[serde(rename_all = "camelCase")]`. If not, per-field `#[serde(rename)]` attributes are needed. → Verify with a live response during task 9.2.
- **`Retry-After` not verified for FMP**: It's unknown whether FMP actually sends `Retry-After` on 429. The custom policy falls back to exponential backoff if the header is absent, so this is safe either way.
- **`csv-async` + `tokio-util` deps**: Both well-maintained; `tokio-util` is likely already a transitive dep. → Check `cargo tree` during implementation.
- **Stream lifetime**: The stream borrows the response body. Rust's type system enforces this — callers can't misuse it.
